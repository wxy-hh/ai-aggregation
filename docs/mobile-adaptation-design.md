# 移动端适配设计文档

> AI 聚合平台 `apps/web` 主应用移动端适配方案

## 1. 文档目标

本文档用于指导 `apps/web` 主应用的移动端适配改造，目标是让前端开发能够基于统一规则完成响应式改造，而不是逐页零散修补断点样式。

本次设计聚焦以下目标：

- 建立全局统一的移动端布局规范
- 明确桌面多栏页面在移动端的收敛策略
- 为核心页面提供可直接落地的改造方案
- 为后续拆分开发任务提供结构化依据

本文档主要服务前端开发落地，因此会优先描述布局、组件、交互、断点、验收和实施顺序，不展开过多产品背景描述。

---

## 2. 适配范围

本次移动端适配范围为整个 `apps/web` 主应用，包括但不限于：

- `app/page.tsx` 首页
- `app/chat/page.tsx` 对话页
- `app/image/page.tsx` 图像生成页
- `app/video/page.tsx` / `video-editor` 视频生成页
- `app/voice/page.tsx` 语音转写页
- `app/history/page.tsx` 历史记录页
- `app/resume/**` 简历相关页面
- `app/destiny/**` 命理相关页面

不在本次设计中展开的内容：

- 后端 API 改造
- Store 结构性重构
- 业务流程重写
- 独立 App 的信息架构调整

本次改造以“最小正确改动”为原则，优先解决结构可用性和核心链路可达性。

---

## 3. 现状问题

当前 `apps/web` 的页面整体偏桌面优先，存在以下共性问题：

### 3.1 全局壳层偏桌面

当前 `AppLayout` 为固定左侧边栏 + 右侧内容区结构：

- 左侧 `GlobalSidebar` 固定占位
- 内容区高度强依赖 `h-screen`
- 页面内部大量使用 `overflow-hidden`

这会导致移动端缺少可用的全局导航模型，也容易引发多层滚动和内容压缩问题。

### 3.2 页面布局普遍依赖多栏工作台

多个核心页面使用桌面工作台模式：

- `chat`：会话管理 + 消息区 + 输入区
- `voice`：主内容区 + 右侧 `RecordingLibrary`
- `image`：左参数栏 + 右结果区
- `video`：左配置栏 + 中预览区 + 右资源栏

这些布局直接缩放到移动端时，会出现核心内容区被挤压、重要辅助功能不可达、交互密度过高等问题。

### 3.3 移动端交互入口缺失

当前部分关键功能在移动端没有对等入口或直接被隐藏：

- `voice` 页的历史记录栏只在 `lg` 显示
- 桌面侧栏承载的应用导航无法直接映射到手机
- 参数面板、资源面板、会话列表等依赖桌面侧栏展示

### 3.4 高度与滚动策略不统一

当前代码中大量存在以下模式：

- 最外层 `h-screen`
- 父容器 `overflow-hidden`
- 子区域单独滚动
- 多个兄弟区域同时设置滚动

这会在手机上导致：

- 内容高度计算不稳定
- 输入框、抽屉、软键盘互相遮挡
- 页面滚动行为不一致

## 4. 设计目标

### 4.1 总体目标

将 `apps/web` 从“桌面工作台集合”改造为“移动端可用、平板可过渡、桌面端保持完整能力”的响应式应用。

### 4.2 具体目标

- 核心任务在手机上可完成
- 桌面端多栏布局在移动端自动收敛为单列主流程
- 辅助侧栏统一抽屉化或面板化
- 全局导航在移动端始终可达
- 页面结构和交互模式保持一致
- 桌面现有能力尽量不倒退

### 4.3 非目标

以下内容不作为本轮移动端适配目标：

- 在手机上 1:1 复刻所有桌面编辑能力
- 首轮即完成所有复杂页面的视觉精修
- 重做现有业务组件的数据流
- 引入大型新状态架构

---

## 5. 断点与响应式策略

建议在整个 `apps/web` 中统一三层断点策略：

- `mobile`：`< 768px`
- `tablet`：`768px ~ 1023px`
- `desktop`：`>= 1024px`

### 5.1 各断点的布局原则

#### mobile

- 单列主流程优先
- 辅助信息通过抽屉、底部面板、分步流程进入
- 不保留桌面三栏布局

#### tablet

- 允许局部双栏
- 可以保留部分辅助区，但不恢复完整工作台
- 作为桌面和手机之间的过渡形态

#### desktop

- 保留当前多栏工作台能力
- 继续支持固定侧栏、分区操作和较高信息密度

### 5.2 实现原则

- 不鼓励每页独立定义断点规则
- 所有页面都优先通过统一容器和统一响应式组件实现适配
- 允许视觉样式因页面而异，但结构规则应一致

## 6. 整体架构设计

### 6.1 `AppLayout` 重构方向

当前 `AppLayout` 默认是桌面结构，不适合直接作为移动端全局壳层。建议将其升级为响应式壳层：

#### desktop

- 保留 `GlobalSidebar`
- 内容区为主要工作区

#### mobile / tablet

- 顶部显示 `MobileHeader`
- 底部显示 `MobileBottomNav`
- 通过 `MobileAppDrawer` 提供更多入口
- 页面内容区自动处理顶部和底部避让

建议拆分为以下结构：

- `AppLayout`
- `DesktopSidebar`
- `MobileHeader`
- `MobileBottomNav`
- `MobileAppDrawer`

页面本身只负责渲染内容，不再自行承担全局导航职责。

### 6.2 全局导航策略

当前 `GlobalSidebar` 承载以下能力：

- 首页入口
- 应用导航
- 固定应用
- 历史入口
- 主题切换
- 应用发现入口

移动端不建议保留“固定到侧边栏”的交互模型，建议重新映射：

#### 移动端底部导航保留高频入口

- 首页
- 对话
- 图像
- 语音
- 更多

#### “更多”抽屉承载低频或扩展入口

- 视频
- 模板
- 历史
- 简历
- Destiny
- 主题切换
- 常用应用管理

即：桌面端保留可定制导航，移动端改为固定高频导航 + 扩展抽屉。

### 6.3 页面容器规范

建议新增统一页面容器层，避免每页重复定义头部、滚动区和边距。

推荐抽象：

- `PageShell`：页面最外层，负责滚动、安全区、背景
- `PageHeader`：统一标题、副标题、主操作区
- `PageBody`：页面内容区，支持单列和工作台布局
- `ResponsivePanel`：桌面为侧栏，移动端为抽屉
- `StickyActionBar`：移动端底部吸附操作栏

统一规则：

- 移动端主内容横向内边距统一为 `px-4`
- 平板统一为 `px-5` 或 `px-6`
- 桌面按页面类型采用 `px-6 ~ px-8`
- 页面标题允许换行，不使用硬编码单行布局
- 主要操作区在手机上可以换行或下移，不强行横向压缩

### 6.4 面板与抽屉策略

桌面页面中大量关键能力位于左栏或右栏，移动端统一转换规则如下：

- 桌面侧栏继续保留
- 移动端复用相同面板内容，放入 `Sheet` / `Drawer`
- 抽屉入口统一放在页面头部或页面内工具栏
- 抽屉高度建议控制在 `70vh ~ 85vh`
- 抽屉内容独立滚动，关闭后保留面板状态

典型映射：

- `RecordingLibrary` -> 移动端历史抽屉
- `image` 参数区 -> 移动端参数抽屉
- `video` 配置区 / 资源区 -> 移动端双抽屉
- `chat` 会话列表 -> 移动端会话抽屉

### 6.5 表单与键盘适配规则

移动端输入体验要求统一：

- 所有主要按钮高度不低于 `44px`
- 所有输入区支持触控尺寸
- 底部输入框优先使用 `sticky bottom-0`
- 键盘弹起时输入区必须可见
- 确认操作按钮尽量固定在弹层底部，不随内容滚动消失

### 6.6 滚动与高度策略

本轮适配必须统一滚动模型，避免继续叠加局部滚动。

统一原则：

- 页面优先只有一个主滚动容器
- 消息流、历史列表、抽屉内容等局部区域才允许内部滚动
- 尽量减少最外层 `h-screen`
- 优先使用 `min-h-dvh` 或 `h-dvh`
- 底部导航存在时，必须给页面内容预留安全区和导航高度

### 6.7 响应式能力层

建议新增轻量工具能力，供组件和页面共享：

- `useBreakpoint()`
- `useIsMobile()`
- `useSafeAreaInsets()`

作用：

- 减少页面内手写 `window.innerWidth` 判断
- 统一移动端分支逻辑
- 处理刘海屏、底部手势条等安全区问题

## 7. 页面分类与改造原则

建议先将页面按交互模式分类，再应用统一改造方式。

### 7.1 对话工作台型

适用页面：

- `chat`
- `voice`

改造原则：

- 保留主流程内容区
- 会话、历史、模式切换等辅助区抽屉化
- 底部输入或操作区吸附

### 7.2 创作编辑型

适用页面：

- `image`
- `video`
- `resume/template`

改造原则：

- 主预览或主结果优先
- 参数区从常驻侧栏改为按需进入
- 复杂桌面三栏收敛为“预览 + 操作面板 + 资源入口”

### 7.3 内容列表型

适用页面：

- `history`
- `templates`
- `home`

改造原则：

- 列表和卡片单列优先
- 搜索和筛选上下堆叠
- 弹窗优先全屏或大底部面板

### 7.4 视觉沉浸型

适用页面：

- `destiny`

改造原则：

- 优先保证结构可用
- 复杂装饰效果可降级
- 右栏知识面板、报告面板统一抽屉化

## 8. 页面级落地方案

## 8.1 对话页 `app/chat/page.tsx`

### 当前问题

- 桌面会话管理结构较重
- 消息区和输入区高度依赖桌面布局
- 模型切换与会话切换在移动端缺少合适入口
- 输入法、键盘和底部输入区可能互相遮挡

### 移动端布局方案

采用“三段式”结构：

- 顶部：标题 + 会话入口 + 模型入口
- 中部：消息流主区域
- 底部：吸附输入区

### 关键改造点

- 会话列表改为移动端抽屉
- 模型选择器改为底部面板或抽屉
- 输入区改为 `sticky bottom-0`
- 欢迎态卡片改为单列轻量布局
- 底部保留发送、附件、停止生成等高频操作

### 移动端必须可完成的操作

- 新建会话
- 切换会话
- 发送消息
- 切换模型
- 重新生成
- 查看长消息内容

### 桌面端保留内容

- 保留完整会话管理结构
- 保留较高信息密度的桌面布局

---

## 8.2 语音页 `app/voice/page.tsx`

### 当前问题

- `RecordingLibrary` 仅在 `lg` 显示
- 实时录音和上传音频共用页面，但移动端结构较拥挤
- 波形区、结果区、操作区在小屏中容易冲突

### 移动端布局方案

采用“模式主流程 + 历史抽屉”结构：

- 顶部：标题 + 模式切换 + 历史入口
- 中部：实时录音或上传音频的主内容区
- 底部：录音操作条或转写结果操作条

### 关键改造点

- `RecordingLibrary` 改为可复用响应式面板
  - 桌面：右侧固定栏
  - 移动端：底部抽屉
- 实时录音和上传音频结果统一为单列结果流
- 历史记录点击后关闭抽屉并恢复主内容
- 录音状态区简化为顶部状态条

### 移动端必须可完成的操作

- 切换实时录音 / 上传音频模式
- 上传音频并查看转写结果
- 查看历史记录
- 恢复历史记录
- 结束实时录音并保存

### 备注

移动端首版不保留桌面右栏常驻形态，历史记录统一按需展开。

---

## 8.3 图像生成页 `app/image/page.tsx`

### 当前问题

- 页面依赖左侧固定参数栏
- 主结果区在小屏幕下会被压缩
- 参数输入、风格切换、结果展示争夺首屏空间

### 移动端布局方案

采用“结果优先 + 参数抽屉”结构：

- 顶部：标题 + 参数入口 + 扩展入口
- 中部：生成结果预览区
- 下方：提示词输入 + 生成按钮
- 参数、风格、负面提示词进入抽屉或折叠区

### 关键改造点

- 左侧设置区不再移动端常驻
- 提示词输入保留在主页面
- 风格选择器改为横向滚动 chips 或简化网格
- 高级参数放入折叠面板或抽屉
- 生成历史缩略图放在结果区下方

### 移动端必须可完成的操作

- 输入提示词
- 切换风格
- 调整关键参数
- 发起生成
- 查看结果
- 下载结果

### 设计原则

移动端第一屏优先展示“输入 + 生成 + 当前结果”，不优先展示完整控制面板。

---

## 8.4 视频生成页 `app/video/**`

### 当前问题

- 当前是典型三栏工作台
- 左配置栏、右资源栏与中间预览竞争空间
- `TimelineBar` 在小屏上不适合桌面式展示

### 移动端布局方案

采用“分步工作流”结构：

- 顶部：返回、标题、资源入口
- 中部：预览区优先
- 下方：时间轴或状态区
- 配置区通过抽屉进入
- 资源区通过独立抽屉进入

### 关键改造点

- `ConfigPanel` 改造为响应式面板
- `AssetsSidebar` 改造为资源抽屉
- `TimelineBar` 移动端支持横向滚动或简化展示
- 生成按钮放入底部固定操作栏
- 预览区优先级高于配置区

### 移动端首版重点能力

- 输入提示词
- 配置关键参数
- 上传参考图
- 发起生成
- 查看生成结果

### 可接受的首版降级

- 不追求完整桌面编辑器体验
- 时间轴能力可适当收缩
- 高级资源管理功能可后移

---

## 8.5 历史页 `app/history/page.tsx`

### 当前问题

- 筛选栏在小屏容易拥挤
- 卡片网格横向密度较高
- 预览弹窗在移动端不适合保持桌面式布局

### 移动端布局方案

采用“顶部筛选 + 单列卡片流”结构：

- 顶部：标题 + 统计信息
- 第二行：类型切换 tabs
- 第三行：搜索框与筛选按钮
- 内容区：单列卡片流

### 关键改造点

- tabs 支持横向滚动
- 搜索与筛选改为上下堆叠
- 历史卡片统一单列
- 图片预览优先全屏弹层或大底部面板展示

### 移动端必须可完成的操作

- 搜索
- 分类切换
- 删除
- 预览
- 跳转回原功能页

---

## 8.6 首页 `app/page.tsx`

### 改造原则

- 首屏优先展示高频功能入口
- 大型装饰区缩短高度，避免首屏信息浪费
- 模块入口卡片改为 1 列或 2 列网格
- 不使用桌面营销式大留白布局

---

## 8.7 简历页 `app/resume/**`

### 改造原则

- 复用现有已存在的移动端适配思路
- 纳入统一壳层、统一抽屉、统一安全区规则
- 编辑区与预览区在移动端继续使用 tab 或抽屉切换
- 不与全局导航和底部操作条冲突

---

## 8.9 Destiny 页面 `app/destiny/**`

### 改造原则

- 优先保证结构可用性
- 复杂视觉装饰可降级
- 报告右栏、知识面板、AI Copilot 面板统一抽屉化
- 大型沉浸式布局在移动端转为上下流式结构

## 10. 样式与交互规范

### 10.1 尺寸规范

- 主要按钮高度不低于 `44px`
- 主要输入框高度不低于 `44px`
- 可点按图标按钮建议为 `40px ~ 44px`
- 页面横向默认间距在移动端统一为 `16px`

### 10.2 文本与标题规范

- 页面标题允许自动换行
- 副标题在移动端可隐藏部分低优先级描述
- 长标签和长标题必须允许折行，不使用硬编码截断替代布局修正

### 10.3 弹层规范

- 移动端详情、筛选、设置优先使用抽屉
- 复杂内容优先全屏弹层
- 抽屉内操作按钮尽量固定底部

### 10.4 列表与卡片规范

- 移动端列表优先单列
- 卡片间距统一，避免视觉拥挤

## 14. 风险与约束

### 14.1 主要风险

- 现有页面高度依赖固定布局，改动时容易引起连锁影响
- `video` 与 `destiny` 页面复杂度高，移动端需要接受首版能力收敛
- 如果不先统一壳层和容器，后续页面会继续各自实现响应式逻辑

### 14.2 约束原则

- 先做结构可用，再做视觉精修
- 尽量复用现有业务组件，不重写核心业务逻辑
- 不在首轮适配中引入大规模 Store 重构
- 对复杂页面允许首版降级，但不允许核心流程不可用

---

## 15. 推荐结论

本项目移动端适配不应以“逐页补断点样式”的方式推进，而应采用“统一壳层 + 页面分类改造 + 核心页面优先”的方案。

推荐执行顺序：

1. 先完成全局壳层与导航体系重构
2. 再完成 `chat / voice / image / history` 四个高频页面改造
3. 最后处理 `video / resume / destiny` 等复杂工作台页面

这样可以在控制风险的前提下，以较小的结构成本获得更稳定的移动端可用性提升。
