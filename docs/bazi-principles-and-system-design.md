# AI 八字测算：原理与系统设计

## 概述

本项目实现了一套完整的 AI 驱动八字命理分析系统。与传统的"本地排盘算法 + 规则式断命"不同，本项目**完全通过大语言模型（火山引擎 ARK 豆包大模型）进行端到端的命理推演**，从用户输入生辰信息到输出完整命盘报告，均由 AI 模型完成。

---

## 一、八字基本原理

### 1.1 天干地支

八字（又称四柱命理）的核心理论基础是天干地支纪时系统：

**十天干**：甲、乙、丙、丁、戊、己、庚、辛、壬、癸
**十二地支**：子、丑、寅、卯、辰、巳、午、未、申、酉、戌、亥

天干与地支两两搭配，组成干支纪年、纪月、纪日、纪时。每个干支组合对应一种五行属性（金木水火土）。

### 1.2 四柱

"八字"即四个时间维度的天干地支组合，称为**四柱**：

| 柱位             | 代表   | 命理含义                       |
| ---------------- | ------ | ------------------------------ |
| **年柱**         | 年干支 | 外在环境、早年基调、祖上根基   |
| **月柱**         | 月干支 | 事业根基、社会关系、成长环境   |
| **日柱（日主）** | 日干支 | **命主本人**，是整个八字的中心 |
| **时柱**         | 时干支 | 行动方式、晚年趋势、子女缘     |

每柱包含一个天干和一个地支，四柱共八个字，故称"八字"。

- 日柱的天干即"**日主**"，代表命主自身，是五行旺衰、十神关系的分析原点
- 年柱以立春为界（非正月初一）
- 月柱以节气为界（如立春→惊蛰为寅月）
- 时柱按出生时辰的地支确定

### 1.3 五行生克

八字分析的核心框架是**五行（金木水火土）**的相生相克关系：

**相生**：木 → 火 → 土 → 金 → 水 → 木（循环相生）
**相克**：木 → 土 → 水 → 火 → 金 → 木（循环相克）

- 五行"生"者为"印"（生我）；"克"者为"官"（克我）
- "我生"者为"食伤"；"我克"者为"财"
- 同五行者为"比劫"

### 1.4 十神体系

十神是以日主（日柱天干）为中心建立的十种关系分类，是八字断命的主要工具：

| 十神             | 关系           | 象征                       |
| ---------------- | -------------- | -------------------------- |
| **正官**         | 克我且阴阳相反 | 责任、规则、事业、名誉     |
| **偏官（七杀）** | 克我且阴阳相同 | 压力、挑战、魄力、竞争     |
| **正印**         | 生我且阴阳相反 | 学识、贵人、长辈、庇护     |
| **偏印（枭神）** | 生我且阴阳相同 | 独特思维、偏门才能         |
| **比肩**         | 同我且阴阳相同 | 自我、朋友、兄弟、竞争     |
| **劫财**         | 同我且阴阳相反 | 同辈、社交、消耗、合作     |
| **食神**         | 我生且阴阳相同 | 享受、才华、表达、福气     |
| **伤官**         | 我生且阴阳相反 | 才华外露、叛逆、创新       |
| **正财**         | 我克且阴阳相反 | 正职收入、稳定财富、配偶   |
| **偏财**         | 我克且阴阳相同 | 意外之财、投资、父亲、社交 |

### 1.5 旺衰格局

"身强"与"身弱"取决于日主在四柱中得到的生扶（印星）和同助（比劫）之多寡：

- **身强**：日主得令（月柱地支为印比）、得地（地支有根）、得势（印比众多）
- **身弱**：日主失令、失地、失势，但若有格局形成（如从格），则为特殊命格

身强喜克泄耗（官杀、食伤、财），身弱喜生扶（印星、比劫），此为调候用神的核心逻辑。

## 三、核心数据流

### 3.1 两阶段流式生成

系统采用**快速阶段 + 完整报告阶段**两阶段策略，兼顾首屏体验和完整深度：

```
Stage 1: 快速阶段 (20s 超时)
┌──────────────┐
│ 生成首屏 3 个  │ → SSE → 前端立即渲染
│ 区块          │       - profileOverview（基本信息）
│              │       - modulesOverview（五模块概览）
│              │       - timeline（流年）
└──────────────┘

Stage 2: 完整报告 (300s 超时)
┌──────────────┐
│ 补充完整 5 个   │ → SSE → 逐区块替换/补充
│ 区块          │       - pillars（四柱）
│              │       - elements（五行）
│              │       - tenGods（十神）
│              │       + profile/modules/timeline 完整版
└──────────────┘
```

**锁区合并机制**：前端维护 `lockedSections` 集合，一旦某个 section 已渲染（无论来自快速阶段还是完整阶段），后续相同 section 不再覆盖，保证界面不闪动。

### 3.2 SSE 流式事件格式

```typescript
// 状态事件
{ type: 'status', status: 'queued' | 'charting' | 'analyzing' | 'finalizing' }

// 区块完成事件
{
  type: 'section-final',
  sectionKey: 'profileOverview' | 'pillars' | 'elementsAndTenGods' | 'modulesOverview' | 'timeline',
  payload: { ... } // 对应类型的数据
}

// 完整报告事件
{ type: 'complete', report: DestinyReport }

// 错误事件
{ type: 'error', error: string }
```

### 3.3 数据模型

```typescript
// 核心命盘报告
DestinyReport {
  profile:    DestinyProfile    // 命主信息（姓名、性别、生辰）
  pillars:    BaZiPillar[]      // 四柱（年/月/日/时各一柱）
  tenGods:    TenGod[]          // 十神（4项含权重值）
  elements:   FiveElement[]     // 五行（金木水火土各一项，0-100分）
  modules: {                    // 五大分析模块
    personality                // 性格底色与优势
    career                     // 事业发展潜力
    love                       // 感情模式与关系
    wealth                     // 财运结构
    health                     // 健康关注点
  }
  timeline:   TimelineItem[]    // 流年（当前起未来3年）
  ziweiPalaces?: ZiweiPalace[] // 紫微12宫（跨模块共享）
  ziweiCenter?: ZiweiCenterInfo
}

// 四柱结构
BaZiPillar {
  stem:    string         // 天干（如 "甲"）
  branch:  string         // 地支（如 "子"）
  label:   string         // 柱名（"年柱"/"月柱"/"日柱"/"时柱"）
  element: FiveElementKey // 五行属性
  tooltip: string         // 解读
}

// 五行
FiveElement {
  key:   FiveElementKey  // 'metal'|'wood'|'water'|'fire'|'earth'
  label: string          // 中文名
  value: number          // 0-100 能量评分
}

// 十神
TenGod {
  key:     string    // 如 'piancai', 'zhengguan'
  label:   string    // 中文名如 '偏财', '正官'
  value:   number    // 0-100 权重
  tooltip: string    // 解读
}

// 流年
TimelineItem {
  year:    number     // 年份
  title:   string     // 标题
  summary: string     // 概述
  detail: {
    opportunities: string[]  // 机遇
    risks:        string[]  // 风险
    actions:      string[]  // 建议行动
  }
}
```

## 四、AI 提示词工程

### 4.1 用户提示词 (User Prompt)

将表单数据映射为结构化提示：

```
请基于以下用户信息生成完整命理报告（中文）：
重要：以下出生日期与出生时间均为农历（阴历）口径，不是公历（阳历）。
姓名：{name}
性别：{gender}
出生日期：{year}-{month}-{day}
出生时间：{hour}:{minute}
出生地：{location}
```

注意关键设计：**声明输入为农历口径**，引导模型按农历进行传统的天干地支排盘，而非按公历日期换算。

### 4.2 系统提示词 (System Prompt)

快速阶段和完整阶段使用不同的 System Prompt：

| 阶段 | 要点                                          | 输出长度    | 温度 |
| ---- | --------------------------------------------- | ----------- | ---- |
| 快速 | 仅含3个区块，summary简洁，bullets 2-3条       | 1600 tokens | 0.2  |
| 完整 | 含全部字段，每项summary 50-90字，bullet ≤18字 | 2400 tokens | 0.3  |
| 重试 | 与完整版类似，但更紧凑                        | 5000 tokens | 0.2  |

核心指令要求：

1. **严格输出 JSON 对象**，禁止额外文字
2. **不输出思考过程**，只返回最终 JSON
3. 五行值为 0-100 数字
4. 流年固定为当年起未来3年
5. 语气稳健，不夸大确定性
6. 内容可执行，避免空泛措辞

### 4.3 容错重试机制

当模型输出被截断（`incomplete_details.reason === 'length'`）时，自动切换到紧凑版 System Prompt 并以更大的 max_output_tokens（5000）重试。若重试仍被截断，则向上层抛出错误。

## 五、数据归一化层 (Normalizer)

模型返回的原始 JSON 经 `normalizeDestinyReport()` 进行多层容错处理：

### 5.1 字段名称兼容

处理同一字段的不同命名：

- `genderLabel` / `gender`
- `birthText` / `birth`
- `locationText` / `birthPlace`
- `name` / `content`（模块名）

### 5.2 四柱格式兼容

模型可能返回两种格式：

- **对象数组**：标准 `{ stem, branch, label, element, tooltip }`
- **字符串数组**：如 `["甲子", "丙寅", "戊辰", "庚申"]` → 自动拆分为天干+地支，按位置赋予柱名

### 5.3 五行归一化

处理两种可能的返回格式：

- **对象格式**：`{ 金: 80, 木: 60, 水: 70, 火: 30, 土: 50 }`
- **数组格式**：`[{ key: 'metal', value: 80 }, ...]`

统一转换为标准五维数组，缺失的五行默认值为 20。

### 5.4 十神归一化

模型可能返回字符串列表或对象数组。字符串格式按出现频率统计权重分布，统一转换为 `{ key, label, value, tooltip }` 数组。若模型未返回十神，使用 4 个默认项（偏财、正财、正官、食神）。

### 5.5 模块归一化

当模型返回数组而非对象时（如 `[{name, content}, ...]`），按位置映射到 `personality / career / love / wealth / health` 五个模块。每个模块提供带默认值的 `safeModule()` 兜底函数。

### 5.6 流年归一化

保证始终返回 3 年数据（当年起），对缺失的 opportunity / risk / action 提供通用默认值。

## 七、AI Copilot（对话式解读）

除了直接生成报告，系统还提供**AI 命理解读助手**：

- 基于已生成的报告摘要进行问答交互
- 报告摘要作为上下文注入，用户可追问细节
- 8 秒超时限制（适配 Vercel 免费层限制）
- Hooks: `ai-usage.ts` 记录用量（`feature: destiny, action: destiny-copilot`）

System Prompt："你是命理报告解读助手。基于给定报告摘要，回答用户关于命盘的提问。给出清晰可执行建议，避免绝对化结论。"

## 八、技术关键点

### 8.1 模型选择

- **火山引擎 ARK / 豆包 doubao-seed-2-0-lite-260428**
- Reasoning Effort: `low`（八字命理不需要复杂推理，重点是结构化输出）
- JSON 输出模式：`text.format.type = 'json_object'`

### 8.2 错误分类

前端对错误的友好分类展示：

| 错误类型     | 触发条件   | 提示文案             |
| ------------ | ---------- | -------------------- |
| `validation` | 400 / 422  | 参数错误，请检查输入 |
| `timeout`    | 408 / 504  | 模型推演超时，请重试 |
| `model`      | 429 / 500+ | 服务暂不可用，请重试 |
| `unknown`    | 其他       | 系统异常             |

### 8.3 AI 用量记录

每次 AI 调用后异步记录用量，支持后续的成本分析和模型调优：

- `feature: 'destiny'`
- `action: 'destiny-report'`
- `provider: 'doubao'`
- `model: 'doubao-seed-2-0-lite-260428'`
- 记录 token 用量

## 九、与传统排盘方案对比

| 维度         | 传统方案（本地算法）         | 本方案（AI 驱动）         |
| ------------ | ---------------------------- | ------------------------- |
| 天干地支排盘 | 需要完整的万年历 + 节气算法  | 由 LLM 内部知识完成       |
| 五行计算     | 按日主 + 月令 + 地支藏干推算 | LLM 直接输出 0-100 评分   |
| 十神推导     | 日主与各天干按生克规则映射   | LLM 直接输出十神及其权重  |
| 用神喜忌     | 需复杂的身强身弱分析         | LLM 隐式推理后输出建议    |
| 流年推算     | 按大运 + 流年干支关系推算    | LLM 按常识 + 命理逻辑生成 |
| 可解释性     | 步骤明确，可追溯             | 端到端黑箱，依赖模型质量  |
| 维护成本     | 高（万年历、节气、算法复杂） | 低（仅需精调 Prompt）     |
| 结果一致性   | 确定性的算法输出             | 随机性（每次可能不同）    |
| 内容深度     | 限于规则，难以生成个性化建议 | 可根据上下文生成丰富解读  |

## 十、设计取舍与演进方向

### 当前设计取舍

1. **纯 AI 驱动而非混合方案**：当前无本地排盘算法，完全依赖 LLM 知识。优点是实现成本低、解读内容丰富；缺点是推理过程不透明、结果不可复现。

2. **两阶段流式策略**：快速阶段 20s 保证首屏体验，完整阶段 300s 保证深度。SSE 流式 + 锁区合并确保渐进式渲染不闪动。

3. **农历口径声明**：用户输入的公历出生日期在 Prompt 中被声明为"农历口径"，引导模型按传统方法排盘。

### 潜在演进方向

1. **混合架构**：引入本地万年历/排盘算法生成四柱 + 五行旺衰，AI 仅负责解读分析。提高确定性和可复现性。

2. **多模型集成**：不同模型负责不同环节（如一个模型排盘、另一个模型解读），利用各自的优势。

3. **知识库增强**：接入专业命理知识库，减少 LLM 的"幻觉"问题，提高专业准确性。

4. **用户反馈闭环**：收集用户对报告的满意度反馈，持续优化 Prompt 和模型选择。

5. **精细化交互**：提供更多的交互式探索（如调整某柱后的重新分析、针对特定模块的追问）。
