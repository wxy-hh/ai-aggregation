# 点击选中问题修复

## 🐛 问题描述

**用户反馈**：

> "点击当前段落区域，选中的还是上边那个"

**问题分析**：
点击某个段落时，高亮显示的是上一个段落，而不是点击的那个。

## 🔍 问题根源

### 根本原因：音频自动更新覆盖用户点击

```typescript
// 音频播放时会持续更新 activeSegmentId
const handleTimeUpdate = () => {
  setCurrentTime(audio.currentTime);

  // 找到当前播放的片段
  const currentSegment = segments.find(
    (seg) => audio.currentTime >= seg.startTime && audio.currentTime <= seg.endTime
  );
  setActiveSegmentId(currentSegment?.id || null); // ⚠️ 覆盖用户点击
};
```

**问题流程**：

```
1. 用户点击第3个段落
   ↓
2. setActiveSegmentId('3')
   ↓
3. 音频跳转到第3个段落的时间
   ↓
4. 音频播放中...
   ↓
5. handleTimeUpdate 触发（每秒多次）
   ↓
6. 发现当前时间在第2个段落范围内（因为时间戳不准确）
   ↓
7. setActiveSegmentId('2')  ← 覆盖了用户的点击
   ↓
8. 用户看到第2个段落高亮 ❌
```

### 为什么会选中上一个段落？

**时间戳不准确导致的偏差**：

```
实际音频时间轴：
├─────────┼─────────┼─────────┼─────────┤
0秒      30秒     60秒     90秒    120秒

估算的时间戳：
├─────────┼─────────┼─────────┼─────────┤
段落1     段落2     段落3     段落4
0-25秒   25-50秒  50-75秒  75-100秒

用户点击段落3：
- 跳转到 50秒
- 但实际音频在 50秒时还在说段落2的内容
- 所以 handleTimeUpdate 检测到当前在段落2的范围
- 高亮显示段落2 ❌
```

## ✅ 修复方案

### 方案：添加用户点击标记

**核心思路**：

- 用户点击时，设置一个标记 `userClickedSegment = true`
- 在标记为 `true` 期间，禁止音频自动更新 `activeSegmentId`
- 3秒后自动恢复，允许音频自动更新

### 实现代码

#### 1. 添加状态标记

```typescript
const [userClickedSegment, setUserClickedSegment] = useState(false);
```

#### 2. 修改音频更新逻辑

```typescript
const handleTimeUpdate = () => {
  setCurrentTime(audio.currentTime);

  // 只有在用户没有手动点击时，才自动更新 activeSegmentId
  if (!userClickedSegment) {
    const currentSegment = segments.find(
      (seg) => audio.currentTime >= seg.startTime && audio.currentTime <= seg.endTime
    );
    setActiveSegmentId(currentSegment?.id || null);
  }
};
```

#### 3. 修改点击处理

```typescript
onClick={(e) => {
  e.stopPropagation();
  setActiveSegmentId(segment.id);
  setUserClickedSegment(true); // 标记用户点击
  handleSeek(segment.startTime);

  // 3秒后恢复自动更新
  setTimeout(() => {
    setUserClickedSegment(false);
  }, 3000);
}}
```

#### 4. 播放结束时重置

```typescript
const handleEnded = () => {
  setIsPlaying(false);
  setUserClickedSegment(false); // 播放结束后重置
};
```

## 🔍 问题排查

### 如果点击后仍然选中错误

**检查点1：查看 userClickedSegment 状态**

在浏览器控制台输入：

```javascript
// 添加调试日志
console.log('userClickedSegment:', userClickedSegment);
```

**检查点2：查看 setTimeout 是否执行**

```typescript
setTimeout(() => {
  console.log('恢复自动更新');
  setUserClickedSegment(false);
}, 3000);
```

**检查点3：查看 activeSegmentId 更新**

```typescript
const handleTimeUpdate = () => {
  console.log('handleTimeUpdate:', {
    userClickedSegment,
    currentTime: audio.currentTime,
    activeSegmentId,
  });

  if (!userClickedSegment) {
    // ...
  }
};
```

### 如果3秒后不恢复自动更新

**可能原因**：

- setTimeout 没有执行
- userClickedSegment 没有重置

**解决方案**：

- 检查是否有多个 setTimeout 冲突
- 使用 useRef 存储 timeout ID，清理旧的 timeout

```typescript
const timeoutRef = useRef<NodeJS.Timeout | null>(null);

onClick={(e) => {
  // 清理旧的 timeout
  if (timeoutRef.current) {
    clearTimeout(timeoutRef.current);
  }

  setUserClickedSegment(true);

  // 设置新的 timeout
  timeoutRef.current = setTimeout(() => {
    setUserClickedSegment(false);
  }, 3000);
}}
```

## 🎯 为什么选择3秒？

### 考虑因素

1. **用户体验**：
   - 太短（1秒）：用户还没看清就被覆盖
   - 太长（10秒）：影响音频自动高亮

2. **音频跳转时间**：
   - 音频跳转需要时间（约0.5秒）
   - 用户需要时间确认（约1-2秒）
   - 留出缓冲时间（约0.5秒）

3. **实际测试**：
   - 3秒是一个平衡点
   - 既能保证用户看清
   - 又不会影响后续操作

### 可以根据需求调整

```typescript
// 短时间锁定（适合快速浏览）
const LOCK_DURATION = 2000; // 2秒

// 长时间锁定（适合仔细阅读）
const LOCK_DURATION = 5000; // 5秒

// 永久锁定（直到用户再次点击）
// 不使用 setTimeout，手动解锁
```

## 📝 相关问题

### 问题1：时间戳不准确

**根本原因**：

- SiliconFlow API 不支持时间戳
- 当前使用估算的时间戳
- 导致音频时间和文本不匹配

**解决方案**：

- 切换到支持时间戳的 API（讯飞、Whisper）
- 或者改进时间戳估算算法

### 问题2：音频识别不全

**根本原因**：

- SiliconFlow API 识别准确率有限
- 某些音频内容被遗漏

**解决方案**：

- 切换到更准确的识别服务
- 提高音频质量
